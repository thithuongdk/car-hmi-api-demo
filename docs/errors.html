Error Codes Reference

All API errors return a consistent JSON structure with error, code, and id fields.

{
  "error": "Short human-readable description",
  "code": 3002,
  "id":   "VAL_MISSING_FIELD"
}
Note: code and id are application-level error codes, independent of HTTP status. Use code (integer) for switch-case in FE logic; use id (string) for logging and tracing.

Optimistic Locking — section_id

section_id is an auto-incrementing integer used to prevent race conditions when multiple clients write simultaneously.

1. Client  GET /api/profile          →  { section_id: 5, ... }
2. Client  PUT /api/profile          →  send { section_id: 5, ... }
3a. OK     section_id matches        →  200 OK, section_id becomes 6
3b. FAIL   Another client PUT first  →  409 { code: 3005, id: "VAL_CONFLICT",
                                             "error": "section_id mismatch: expected 6, got 5" }
4. Handle 409: GET again to fetch new section_id, then retry PUT

Endpoints requiring section_id: PUT /api/profile, PUT /config

Reconnect strategy (exponential backoff)

let retryDelay = 1000;
function connect() {
  const ws = new WebSocket("wss://car-hmi-api-demo.onrender.com/ws/signals");
  ws.addEventListener("open",  () => { retryDelay = 1000; });
  ws.addEventListener("close", (e) => {
    if (e.code !== 1000) {
      setTimeout(() => { retryDelay = Math.min(retryDelay * 2, 30000); connect(); }, retryDelay);
    }
  });
}

Signal Write — Debug Flow

PUT /signals/SomeName  →  404 / 3004  →  Signal not in signal.json
PUT /signals/SomeName  →  403 / 4002  →  TX: false (sensor, cannot write)
PUT /signals/SomeName  →  403 / 4004  →  Vehicle is moving, command blocked
PUT /signals/SomeName  →  422 / 3003  →  Value outside [min, max]
PUT /signals/SomeName  →  202         →  OK, queued for CAN bus

POST /signals/batch_update  →  202  →  Per-signal results:
    { name, value, status: "ok" | "not_writable" | "not_found" | "out_of_range" }
